# 支付网关系统设计

> 版本: v1.0
> 日期: 2026-04-01
> 状态: 已批准

## 1. 项目概述

### 1.1 项目背景
建设一个商业化输出的综合支付平台，支持线上线下全场景支付，具备商户入驻、分账、担保交易、跨境结算等完整能力。

### 1.2 项目目标
- 日交易量支撑百万级
- 7x24 高可用服务
- T+0 实时结算能力
- 完整的风控体系

### 1.3 适用范围
- 多商户入驻与管理
- 线上（电商/APP）与线下（扫码/刷卡）全场景
- 担保交易与分账结算
- 跨境支付支持

## 3. 核心服务设计

### 3.1 服务职责矩阵

| 服务名称 | 核心职责 | 主要接口 |
|----------|----------|----------|
| **商户服务** | 商户入驻、资质认证、签约、配置管理 | 商户注册、认证、签约、配置查询 |
| **交易服务** | 支付、退款、撤销、查询、幂等控制 | 支付、退款、撤销、交易查询 |
| **账务服务** | 账户、流水、分账、余额冻结/解冻 | 账户查询、流水查询、冻结/解冻 |
| **渠道服务** | 渠道接入、路由选择、通道管理 | 渠道路由、通道查询、渠道配置 |
| **结算服务** | 账单生成、对账、提现、跨境结算 | 账单查询、对账确认、提现申请 |
| **风控服务** | 交易风控、反欺诈、限频 | 风控评估、黑名单管理 |
| **通知服务** | 支付结果通知、商户webhook | 通知发送、模板管理 |
| **用户服务** | 平台用户、操作员、权限 | 用户CRUD、权限管理 |

### 3.2 服务间通信

采用**混合模式**：
- **同步调用**：核心链路（支付 → 账务），使用 gRPC
- **异步消息**：非核心链路（通知、分析），使用 Kafka

## 4. 数据层设计

### 4.1 分库分表策略

按**商户ID哈希**分4个库，每库64张表：

```
库1: merchant_db_00 ~ merchant_db_15       # 商户相关
库2: transaction_db_00 ~ transaction_db_15 # 交易流水
库3: account_db_00 ~ account_db_15        # 账户账务
库4: settlement_db_00 ~ settlement_db_15  # 结算账单
```

### 4.2 分片规则

| 表类型 | 分片键 | 分片算法 |
|--------|--------|----------|
| 商户表 | merchant_id | 哈希 |
| 交易表 | merchant_id + create_time | 哈希+范围 |
| 账户表 | account_id | 哈希 |
| 账单表 | merchant_id + bill_date | 哈希 |

### 4.3 核心数据模型

#### 4.3.1 商户 (merchant)
```
merchant_id          # 商户ID (UUID)
merchant_name        # 商户名称
merchant_type        # 商户类型 (个体/企业)
status               # 状态 (审核中/正常/冻结)
资质信息             # 营业执照、法人等
结算信息             # 银行账户、结算周期
费率配置             # 各通道费率
created_at           # 创建时间
updated_at           # 更新时间
```

#### 4.3.2 交易 (transaction)
```
transaction_id       # 交易ID (UUID)
merchant_id         # 商户ID
order_id            # 商户订单号
channel             # 支付渠道 (alipay/wechat/unionpay)
channel_order_id    # 渠道订单号
amount              # 交易金额 (分)
currency            # 币种 (CNY/USD)
status              # 交易状态
payment_method      # 支付方式 (扫码/刷卡/NFC)
scene               # 场景 (线上/线下)
created_at          # 创建时间
completed_at        # 完成时间
```

#### 4.3.3 账户 (account)
```
account_id          # 账户ID (UUID)
merchant_id         # 商户ID
account_type        # 账户类型 (可用/冻结/待结算)
balance             # 余额 (分)
currency            # 币种
version             # 版本号 (乐观锁)
created_at          # 创建时间
updated_at          # 更新时间
```

#### 4.3.4 账务流水 (account_log)
```
log_id              # 流水ID
account_id          # 账户ID
transaction_id      # 关联交易ID
type                # 类型 (收入/支出/冻结/解冻)
amount              # 金额 (分)
balance_before      # 变动前余额
balance_after       # 变动后余额
remark              # 备注
created_at          # 创建时间
```

## 5. 核心交易流程

### 5.1 支付流程 (Saga 编排)

```
┌─────────┐    ┌──────────┐    ┌─────────┐    ┌──────────┐
│  商户   │───▶│ API GW   │───▶│ 交易服务 │───▶│ 风控服务 │
└─────────┘    └──────────┘    └────┬────┘    └──────────┘
                                   │
                    ┌──────────────┼──────────────┐
                    ▼              ▼              ▼
              ┌─────────┐   ┌──────────┐   ┌──────────┐
              │渠道服务 │   │ 账务服务 │   │ 结算服务 │
              │(渠道路由)│   │ (记账)   │   │ (更新账单)│
              └────┬────┘   └────┬─────┘   └────┬─────┘
                   │            │              │
                   ▼            ▼              ▼
              ┌─────────────────────────────────────────┐
              │         Saga Orchestrator              │
              │    (Transaction Coordinator)           │
              └─────────────────────────────────────────┘
```

#### 详细步骤：
1. **请求接收**：API Gateway 接收支付请求，进行基础校验
2. **幂等检查**：交易服务检查订单号是否已存在
3. **风控评估**：风控服务进行交易风控
4. **渠道路由**：渠道服务根据商户配置和通道状态选择最优渠道
5. **通道调用**：调用第三方支付渠道完成支付
6. **Saga 编排**：
   - Step 1: 账务服务记录收入流水
   - Step 2: 结算服务更新待结算金额
   - Step 3: 如有分账，触发分账流程
7. **结果通知**：通知服务推送支付结果至商户

### 5.2 担保交易流程

```
1. 买家付款 → 资金冻结
2. 卖家发货
3. 买家确认收货 → 资金解冻给卖家
4. 如有纠纷 → 进入风控/人工处理
```

### 5.3 分账流程

```
1. 交易完成后，触发分账
2. 按商户配置的比例/金额拆分
3. 各方账户增加可用余额
4. 生成各自的分账流水
```

## 7. 风控设计

### 7.1 风控维度

| 维度 | 说明 |
|------|------|
| **金额风控** | 单笔限额、日累计限额 |
| **频率风控** | 单位时间内交易次数限制 |
| **商户风控** | 商户评分、黑名单 |
| **渠道风控** | 通道风险评估 |
| **行为风控** | 异常模式识别 |

### 7.2 风控规则示例

```
规则1: 单笔金额 > 50000元 → 人工审核
规则2: 同一IP 1分钟内 > 10笔 → 限频
规则3: 商户评分 < 60分 → 延迟结算
规则4: 异地登录 + 大额交易 → 风控警报
```

## 8. 结算设计

### 8.1 结算周期

| 商户类型 | 结算周期 | 说明 |
|----------|----------|------|
| 优质商户 | D+0 (T+0) | 实时结算，收取服务费 |
| 普通商户 | D+1 | 次日结算 |
| 新商户 | D+7 | 风险期延长 |

### 8.2 对账流程

```
1. 每日凌晨生成前一日账单
2. 商户对对账差异发起申诉
3. 风控/客服处理差异
4. 确认后完成结算
```

## 9. 高可用设计

### 9.1 限流与熔断

- Sentinel 用于流量控制、熔断降级
- 核心接口 QPS 限流
- 第三方渠道超时自动切换

### 9.2 数据灾备

- MySQL 主从复制，跨机房部署
- Redis 主从 + 哨兵
- Kafka 消息持久化 + 多副本

### 9.3 监控告警

- 交易成功率
- 渠道调用延迟
- 系统资源使用率
- 异常交易波动

## 10. 安全设计

### 10.1 交易安全

- 支付密码 + 短信验证码
- 数字签名 (RSA)
- HTTPS 传输加密

### 10.2 数据安全

- 敏感数据加密存储 (AES)
- 信用卡信息 Token 化
- 商户密钥隔离存储
